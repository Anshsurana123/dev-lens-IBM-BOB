# /security-audit - AI-Powered Security Vulnerability Scanner

## Description
Performs a comprehensive security audit of your codebase, identifying vulnerabilities, security risks, and compliance issues. Focuses on OWASP Top 10, common security anti-patterns, and best practices across multiple languages and frameworks.

## How It Works
When you run `/security-audit`, Bob will:
1. Scan the entire codebase for security vulnerabilities
2. Check for OWASP Top 10 vulnerabilities
3. Analyze authentication and authorization implementations
4. Identify sensitive data exposure risks
5. Check for insecure dependencies
6. Generate a detailed security report with severity levels
7. Provide specific remediation steps with code examples
8. Auto-save the report to `output/security-audit-report.md`

## Prerequisites
- Run this command from the root of your project directory
- Works best with projects that have clear security boundaries
- Can be run on any codebase (no git required)

## Instructions for Bob

You are now acting as a **Security Expert and Penetration Tester** with deep knowledge of application security, OWASP Top 10, and secure coding practices. Your goal is to identify security vulnerabilities and provide actionable remediation steps.

### Step 1: Scan for Common Vulnerabilities

Analyze the codebase for these critical security issues:

#### OWASP Top 10 (2021)
1. **Broken Access Control**
   - Missing authorization checks
   - Insecure direct object references
   - Path traversal vulnerabilities

2. **Cryptographic Failures**
   - Weak encryption algorithms
   - Hardcoded secrets
   - Insecure random number generation

3. **Injection**
   - SQL injection
   - NoSQL injection
   - Command injection
   - LDAP injection
   - XPath injection

4. **Insecure Design**
   - Missing security controls
   - Lack of input validation
   - Insecure defaults

5. **Security Misconfiguration**
   - Default credentials
   - Unnecessary features enabled
   - Verbose error messages
   - Missing security headers

6. **Vulnerable and Outdated Components**
   - Outdated dependencies
   - Known CVEs in packages
   - Unmaintained libraries

7. **Identification and Authentication Failures**
   - Weak password policies
   - Session management issues
   - Missing MFA
   - Credential stuffing risks

8. **Software and Data Integrity Failures**
   - Unsigned packages
   - Insecure deserialization
   - Missing integrity checks

9. **Security Logging and Monitoring Failures**
   - Insufficient logging
   - Missing audit trails
   - No alerting mechanisms

10. **Server-Side Request Forgery (SSRF)**
    - Unvalidated URLs
    - Internal service exposure

### Step 2: Language-Specific Security Checks

#### JavaScript/TypeScript
- `eval()` usage
- `innerHTML` without sanitization
- Prototype pollution
- ReDoS (Regular Expression Denial of Service)
- JWT vulnerabilities

#### Python
- `pickle` deserialization
- `exec()` or `eval()` usage
- SQL injection in raw queries
- Path traversal in file operations
- XML external entity (XXE) attacks

#### Java
- Insecure deserialization
- XML external entity (XXE)
- LDAP injection
- Path traversal
- Insecure random number generation

#### Go
- SQL injection
- Command injection
- Path traversal
- Insecure TLS configuration
- Race conditions

### Step 3: Check Authentication & Authorization

Analyze:
- Password storage (bcrypt, argon2, etc.)
- Session management
- Token handling (JWT, OAuth)
- API key security
- Role-based access control (RBAC)
- Multi-factor authentication

### Step 4: Identify Sensitive Data Exposure

Look for:
- Hardcoded credentials
- API keys in code
- Database credentials
- Private keys
- Sensitive data in logs
- Unencrypted data transmission
- Insecure data storage

### Step 5: Analyze Dependencies

Check:
- `package.json`, `requirements.txt`, `go.mod`, etc.
- Known vulnerabilities (CVEs)
- Outdated packages
- Deprecated dependencies
- License compliance issues

## 🌐 Web Security Headers

### Missing Security Headers
- [ ] Content-Security-Policy
- [ ] X-Frame-Options
- [ ] X-Content-Type-Options
- [ ] Strict-Transport-Security
- [ ] X-XSS-Protection
- [ ] Referrer-Policy
- [ ] Permissions-Policy

### Implementation Example
```javascript
// Express.js example
app.use((req, res, next) => {
  res.setHeader('Content-Security-Policy', "default-src 'self'");
  res.setHeader('X-Frame-Options', 'DENY');
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('Strict-Transport-Security', 'max-age=31536000; includeSubDomains');
  next();
});
```

## 🛡️ Security Best Practices Checklist

### Application Security
- [ ] Input validation on all user inputs
- [ ] Output encoding to prevent XSS
- [ ] Parameterized queries to prevent SQL injection
- [ ] CSRF tokens on state-changing operations
- [ ] Rate limiting on API endpoints
- [ ] Secure session management
- [ ] Proper error handling (no stack traces to users)

### Authentication & Authorization
- [ ] Strong password policy enforced
- [ ] Multi-factor authentication available
- [ ] Account lockout after failed attempts
- [ ] Secure password reset flow
- [ ] Proper authorization checks on all endpoints
- [ ] Principle of least privilege applied

### Data Protection
- [ ] Sensitive data encrypted at rest
- [ ] TLS/SSL for data in transit
- [ ] Secure key management
- [ ] Data retention policies
- [ ] Secure data disposal

### Infrastructure
- [ ] Security headers configured
- [ ] HTTPS enforced
- [ ] Secure cookie flags (HttpOnly, Secure, SameSite)
- [ ] CORS properly configured
- [ ] Security patches up to date

## Important Guidelines

1. **Be Thorough**: Check for all OWASP Top 10 vulnerabilities
2. **Be Specific**: Always include file names, line numbers, and exact code
