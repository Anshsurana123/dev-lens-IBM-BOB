# /generate-commit - Smart Commit Message Generator

## Description
Analyzes your staged git changes and generates meaningful, conventional commit messages that follow best practices. Saves time and ensures consistent commit history.

## How It Works
When you run `/generate-commit`, Bob will:
1. Read your staged git changes (git diff --staged)
2. Analyze what changed and why
3. Generate a conventional commit message
4. Suggest commit body with detailed changes
5. Provide multiple commit message options
6. Auto-save suggestions to `output/commit-message.txt`

## Usage Examples

```bash
# Generate commit message for staged changes
/generate-commit

# Generate with specific context
/generate-commit --context "Fixing bug #123"

# Generate for specific files
/generate-commit src/auth.js src/utils.js
```

## 🎨 Conventional Commits Format

### Structure
```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Types
- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that don't affect code meaning
- **refactor**: Code change that neither fixes a bug nor adds a feature
- **perf**: Code change that improves performance
- **test**: Adding missing tests or correcting existing tests
- **build**: Changes that affect the build system
- **ci**: Changes to CI configuration files and scripts
- **chore**: Other changes that don't modify src or test files
- **revert**: Reverts a previous commit

### Examples
```
feat(api): add user authentication endpoint
fix(ui): resolve button alignment issue on mobile
docs(readme): update installation instructions
refactor(utils): simplify date formatting logic
test(auth): add unit tests for login flow
```

## 💡 Best Practices Applied

✅ **Clear and Concise**: Subject line under 50 characters  
✅ **Imperative Mood**: "Add feature" not "Added feature"  
✅ **Conventional Format**: Follows conventional commits spec  
✅ **Meaningful Scope**: Indicates affected component  
✅ **Detailed Body**: Explains what and why  
✅ **References Issues**: Links to issue tracker  
✅ **Breaking Changes**: Clearly marked if present

## Tips for Better Commit Messages

1. **Be Specific**: "Fix login bug" → "Fix null pointer error in login validation"
2. **Use Imperative Mood**: "Add" not "Added" or "Adds"
3. **Explain Why**: Don't just say what changed, explain why
4. **Reference Issues**: Always link to issue tracker
5. **Keep It Atomic**: One logical change per commit
6. **Review Before Committing**: Use `git diff --staged` to verify
